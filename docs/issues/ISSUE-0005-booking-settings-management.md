## Agent Brief

**Category:** enhancement
**Summary:** Add typed Booking Settings Management so reservation policy values are configurable instead of hardcoded.

**Current behavior:**
The application has runtime settings for infrastructure concerns such as database URL, token secret, and allowed student email domains. There is no implemented Booking Settings Management workflow yet for Super Admins, no typed booking/deadline settings object for reservation services, and no persisted `SystemSetting`-style source for policy values.

**Desired behavior:**
Super Admins should be able to view and update the MVP booking/deadline settings through public service/API behavior. Reservation-related services should consume these values through a typed settings object rather than raw string key-value lookups. Defaults should match the MVP decisions in the parent PRD: 14-day minimum booking lead time, 60-day maximum advance booking, document/payment deadlines, approval cutoffs, and allowed student email domains. Duration values should be stored internally as hours, even if a later UI presents them as days.

**Key interfaces:**
- Typed booking/deadline settings object — exposes booking lead hours, advance booking hours, document upload/payment deadlines, approval cutoffs, and allowed student email domains with validated typed fields.
- Booking Settings Management service/module — owns reading, updating, defaulting, validation, and conversion rules for booking/deadline settings.
- Super Admin API/service behavior — allows only Super Admin users to view and update booking/deadline settings.
- Reservation policy consumers — should read settings from the typed object rather than hardcoded constants or raw persistence keys.

**Acceptance criteria:**
- [ ] Super Admin can view current booking/deadline settings through a public service/API interface.
- [ ] Super Admin can update booking lead hours, maximum advance booking hours, document upload deadline, payment deadline, final approval cutoff values, and allowed student email domains.
- [ ] Non-Super Admin users cannot view or update admin booking settings.
- [ ] Defaults match the MVP PRD decisions, including 14-day booking lead time and 60-day maximum advance booking represented internally in hours.
- [ ] Settings are exposed to reservation policy code through a typed object, not ad hoc raw key-value lookups.
- [ ] Invalid setting updates return stable validation errors without partially applying changes.
- [ ] Implementation follows the repo TDD rule: add one failing behavior test, implement the smallest passing change, then refactor while green.

**Out of scope:**
- Building the frontend settings screen.
- Implementing reservation creation or full Booking Policy enforcement beyond wiring policy consumers to the typed settings object where needed.
- Payment gateway behavior, deadline worker behavior, or notification delivery.
- Changing the existing runtime infrastructure settings contract except where allowed student email domains must be exposed through the booking settings workflow.
