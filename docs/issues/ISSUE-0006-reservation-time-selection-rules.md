## Agent Brief

**Category:** enhancement
**Summary:** Add the student reservation time-selection path with backend-enforced booking-window rules.

**Current behavior:**
Students can browse facility details and check facility availability against open hours, blackout periods, and existing blocking reservations. Booking settings already express the default lead-time and advance-booking windows. There is not yet a complete student reservation time-selection path that validates all reservation time rules before a reservation creation flow proceeds.

**Desired behavior:**
Students should be able to choose a facility reservation date, start time, and end time. The backend must validate the requested time range as the source of truth and reject invalid choices with user-facing Bahasa Indonesia messages suitable for the reservation time-selection UI/API response.

Validation must be evaluated in the Asia/Jakarta business timezone while persisted timestamps remain UTC. The backend should reject requested ranges that:

- are not aligned to 5-minute increments
- are shorter than one hour
- cross midnight or otherwise span more than one local calendar day
- fall outside the facility open hours
- overlap a facility blackout period
- overlap an active blocking reservation
- start less than the configured minimum booking lead time ahead of the current time, defaulting to 14 days
- start more than the configured maximum booking advance time ahead of the current time, defaulting to 60 days

**Key interfaces:**
- Facility availability checking should remain the public backend contract for open-hour, blackout, and blocking-reservation checks.
- Booking settings should remain the source for configurable lead-time and advance-booking limits.
- Reservation time input/output shapes should accept timezone-aware instants from clients and normalize/persist reservation timestamps as UTC.
- Validation errors should expose stable machine-readable reasons and Bahasa Indonesia user-facing messages.

**Acceptance criteria:**
- [ ] Students can submit or preview a reservation time selection containing facility, date, start time, and end time.
- [ ] Backend validation rejects non-5-minute increments.
- [ ] Backend validation rejects durations shorter than one hour.
- [ ] Backend validation rejects ranges that cross midnight in Asia/Jakarta.
- [ ] Backend validation rejects ranges outside facility open hours.
- [ ] Backend validation rejects ranges overlapping blackout periods.
- [ ] Backend validation rejects ranges overlapping active blocking reservations.
- [ ] Backend validation rejects ranges starting less than 14 days ahead by default, using the configured minimum lead time when changed.
- [ ] Backend validation rejects ranges starting more than 60 days ahead by default, using the configured maximum advance window when changed.
- [ ] Successful validation evaluates business rules in Asia/Jakarta and keeps persisted/returned reservation timestamps normalized consistently with the existing UTC timestamp convention.
- [ ] Invalid time choices return Bahasa Indonesia messages that the student UI can display directly.
- [ ] Tests follow the repo TDD rule: one observable behavior test at a time through public module/API interfaces.

**Out of scope:**
- Payment, approval-letter generation, document upload, or staff review workflows.
- Changing the configured default booking windows unless a test specifically covers using existing settings.
- Reworking facility catalog, organization unit management, or public calendar behavior beyond what the time-selection path needs.
- Database-level exclusion constraints for double-booking unless already needed by the chosen vertical slice; service/API enforcement is sufficient for this issue.
