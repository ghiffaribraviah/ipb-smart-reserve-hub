## Agent Brief

**Category:** enhancement
**Summary:** Build reservation details submission with conflict-protected facility holds.

**Current behavior:**
Students can browse active Facilities, inspect public Facility detail and calendar entries, validate Reservation Time Selection rules, and choose active Organization Units. There is not yet a reservation submission workflow that creates a held reservation from event details.

**Desired behavior:**
A student should be able to submit reservation details after selecting a valid Facility time. The system should create a reservation code, persist the reservation in the initial workflow status, snapshot relevant display data at creation time, and prevent overlapping active blocking reservations for the same Facility. Conflict prevention must be enforced transactionally, not only through a preflight availability check.

**Key interfaces:**
- Reservation creation Module — accepts the current student, Facility, selected time range, activity title, event description, participant count, Organization Unit, and contact phone.
- Reservation persistence seam — stores reservation records and supports conflict detection/transactional hold creation.
- Facility Availability / Reservation Time Selection — should remain the source of booking-time validation before hold creation.
- Organization Unit selection — should only allow active Organization Units for new student-facing reservations.
- Reservation status values — add or use the initial status that represents a held reservation awaiting the next approval-letter step.

**Acceptance criteria:**
- [ ] A student can submit activity title, event description, participant count, Organization Unit, and contact phone for a selected Facility time.
- [ ] The system creates a unique human-readable reservation code.
- [ ] The initial reservation status is persisted and visible through the student's reservation list/detail behavior.
- [ ] Facility price and Organization Unit name are snapshotted on the reservation at creation time.
- [ ] Overlapping active blocking reservations for the same Facility are rejected transactionally.
- [ ] Inactive Facilities and inactive Organization Units cannot be used for new reservation submission.
- [ ] Implementation follows TDD with one behavior test at a time.

**Out of scope:**
- Approval-letter PDF generation and download.
- Signed-letter upload and staff document review.
- Payment receipt upload and payment review.
- Cancellation, notifications, reviews, and deadline workers.
