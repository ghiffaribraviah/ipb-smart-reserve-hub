## Agent Brief

**Category:** enhancement
**Summary:** Generate and serve stable Bahasa Indonesia approval-letter PDFs for student reservations.

**Current behavior:**
Students can create held reservations, receive a human-readable reservation code, and view their reservation list/detail. A new reservation starts in the status that waits for document upload, but the system does not yet generate the prefilled approval letter that students need before uploading a signed copy. The PDF generation and storage packages exist as architectural placeholders, and private letter download behavior is not exposed through the student reservation API yet.

**Desired behavior:**
After a reservation is created, the system should have exactly one stable generated approval letter PDF tied to that reservation and reservation code. Students should be able to access a separate approval-letter workflow surface for their own reservation and download the same generated letter multiple times. The generated PDF must use formal Bahasa Indonesia and include the reservation, student identity, organization unit, facility, event, contact, signature, and internal verification details needed for offline approval. Private downloads must be authorized before returning the file or a short-lived signed download URL through the storage abstraction.

The letter should remain stable for a reservation once generated. Repeated student access should not create different letters or change the stored file identity unless a deliberate regeneration behavior is added in a future issue.

**Key interfaces:**
- Reservation approval-letter workflow service — exposes student-owned letter lookup/download behavior for a reservation.
- Approval-letter PDF generator — creates the Bahasa Indonesia PDF from reservation, student, facility, organization unit, event, contact, signature-area, and internal verification data.
- Private storage abstraction — stores generated letters privately and serves authorized downloads or signed download URLs without exposing raw storage paths.
- Reservation persistence model — records enough generated-letter metadata to make the letter stable and reusable across repeated downloads.
- Student reservation API — exposes workflow-specific approval-letter endpoints separate from general reservation detail.
- Access policy / authorization boundary — ensures a student can only access generated letters for reservations they own.

**Acceptance criteria:**
- [ ] Creating or first accessing a reservation approval letter results in one generated PDF tied to that reservation and reservation code.
- [ ] Repeated student access/download for the same reservation returns the same generated letter metadata/file identity rather than creating duplicate or inconsistent letters.
- [ ] Students can access a separate approval-letter API/page surface for their own reservation.
- [ ] Students can download the generated approval letter multiple times after authorization.
- [ ] A student cannot access or download another student's generated approval letter.
- [ ] Private generated-letter downloads are served through the storage abstraction, not via public storage paths.
- [ ] The generated PDF is in Bahasa Indonesia and includes reservation code, generation date, student identity, organization unit, facility name/location, reservation date/time, activity title, event description, participant count, contact phone/WhatsApp, responsibility statement, signature areas, TU contact, and an internal staff-verification QR/link or equivalent internal verification data.
- [ ] Internal verification links or QR targets require staff or Super Admin authentication; no public verification link is introduced.
- [ ] Implementation follows TDD with one behavior test at a time.

**Out of scope:**
- Signed approval-letter upload.
- Staff document review, approval, and rejection.
- Payment workflow and receipt upload.
- Automated comparison between generated and signed letters.
- Public QR verification links.
- Regenerating letters after reservation data changes.
- Cancellation, notifications, deadline workers, and review workflows.
