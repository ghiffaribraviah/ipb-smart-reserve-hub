## Agent Brief

**Category:** enhancement
**Summary:** Add in-app notifications with read tracking and workflow-triggered reservation messages.

**Current behavior:**
The backend now has reservation submission, approval-letter generation, signed-letter upload/review, payment receipt upload/review, staff facility assignment, deadline worker processing, and completed-status handling. There is no notification model, inbox, read/unread tracking, or workflow-triggered notification creation yet.

**Desired behavior:**
Users should have an in-app notification inbox. Important reservation workflow transitions should synchronously create notification entries for the affected student, assigned facility staff, and Super Admin users as appropriate. Notification entries should support read/unread state through a read timestamp. Staff-facing notifications should be scoped to staff assigned to the reservation facility, plus Super Admin. Student overdue notifications should include enough facility TU contact information for follow-up.

**Key interfaces:**
- Notification domain model — should capture recipient, title/message, optional reservation reference, created timestamp, and read timestamp.
- Notification service/module — should create notifications for reservation workflow transitions and expose inbox/read behavior.
- Reservation workflow services — should call notification creation at important transitions such as reservation creation, document upload, document approval/rejection, payment upload, payment approval/rejection, deadline expiration, overdue verification, cancellation transitions once available, and completion where relevant.
- Staff assignment lookup — should determine assigned staff recipients for facility-scoped staff notifications.
- Super Admin recipient lookup — should include Super Admin users on staff-facing operational notifications.
- HTTP/API routes — should expose user notification inbox and mark-read behavior through authenticated public interfaces.

**Acceptance criteria:**
- [ ] Authenticated users can view their notification inbox entries, newest first.
- [ ] Notification entries expose read/unread state using `read_at` or equivalent read timestamp.
- [ ] Users can mark notifications as read without affecting other users' notifications.
- [ ] Reservation, document, payment, expiration, overdue, and completion transitions create relevant student notifications where applicable.
- [ ] Staff-facing reservation notifications go to assigned facility staff plus Super Admin users.
- [ ] Student overdue verification notifications include facility TU contact information.
- [ ] Notification behavior is tested through public service/API interfaces with TDD.

**Out of scope:**
- Email, WhatsApp, SMS, push notifications, or external delivery providers.
- Notification preferences, digest scheduling, or templates beyond in-app text.
- Real-time websocket delivery.
- Cancellation-specific notifications until the cancellation workflow issue is implemented.
