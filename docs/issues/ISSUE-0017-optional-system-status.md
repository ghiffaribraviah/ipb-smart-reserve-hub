## Agent Brief

**Category:** enhancement
**Summary:** Add an optional Super Admin system status page.

**Current behavior:**
The app has Super Admin access control and a backend application shell, but no read-only operational status view that summarizes runtime health for maintainers.

**Desired behavior:**
Super Admins should be able to open a read-only system status view. The view should report backend health, database connectivity, storage connectivity when storage is configured, app version or build information when available, and worker/scheduler status when available. Non-admin users must not be able to access the status data.

**Key interfaces:**
- Access Policy — should include a named action for viewing system status.
- System status Module — gathers health facts and returns a typed status result for HTTP/UI callers.
- Application Settings — should provide any configured runtime values needed for safe health checks without exposing secrets.
- HTTP Application — should expose the status behavior through an authenticated Super Admin route.

**Acceptance criteria:**
- [ ] Super Admin can access a read-only system status endpoint or page.
- [ ] Non-admin users cannot access system status.
- [ ] Status includes backend health.
- [ ] Status includes database connectivity.
- [ ] Status includes storage connectivity when storage is configured; unavailable storage config should be reported safely rather than crashing the app.
- [ ] Status includes app version/build information when available.
- [ ] Status includes worker/scheduler status when available, or an explicit not-configured/unknown state.
- [ ] Implementation follows TDD with one behavior test at a time.

**Out of scope:**
- Alerting, metrics dashboards, tracing, and log aggregation.
- Deployment provisioning itself.
- Making optional system status a blocker for core reservation workflows.
