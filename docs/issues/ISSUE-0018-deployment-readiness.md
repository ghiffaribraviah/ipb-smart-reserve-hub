## Agent Brief

**Category:** enhancement
**Summary:** Prepare the backend/API for production-like demo deployment.

**Current behavior:**
The repository currently contains a FastAPI backend with service/API tests and runtime settings, but backend deployment readiness is not yet complete. The app still defaults to local development concerns such as SQLite, a development secret key, in-memory private storage, schema creation through runtime metadata, and status checks that report storage and worker as not configured. There is no committed production deployment entrypoint/configuration, production environment variable reference, migration/deployment documentation, or smoke-check path that demonstrates the backend is reachable in a demo environment.

The frontend is not ready yet, so frontend/Vercel deployment should not block this issue.

**Desired behavior:**
The backend should be deployable and accessible for a demo on the agreed backend infrastructure, with enough production-like configuration and documentation for an agent or maintainer to run the API against PostgreSQL and verify core health.

The deployed backend/API should expose a clear health or system-status check suitable for demo validation. Production settings should avoid insecure development defaults where deployment requires explicit secrets or service configuration. PostgreSQL configuration and migration/schema setup should be documented or implemented in a way that fits the current backend maturity. Storage and worker/scheduler configuration should be documented clearly, even if the first demo uses a minimal implementation.

**Key interfaces:**
- Application settings/environment configuration — must document and support the required production variables for database connectivity, secret key, allowed student email domains, storage configuration, worker behavior, and public base URLs if needed.
- FastAPI application factory/runtime entrypoint — must remain deployable by an ASGI server and should work with deployment platform process configuration.
- Database setup path — must support PostgreSQL for deployment and document how schema/migrations are executed for the demo environment.
- System status or health endpoint — should let maintainers verify backend availability, database connectivity, storage readiness/configuration status, application version, and worker readiness/configuration status.
- Storage service abstraction/configuration — should distinguish public facility image storage from private reservation files in production configuration or documentation.
- Worker/scheduler process configuration — should document or provide the command/process needed for reservation deadline and completion jobs in production.

**Acceptance criteria:**
- [ ] Backend/API deployment configuration is ready for the agreed backend hosting platform.
- [ ] The backend can run under a production ASGI server with a documented command.
- [ ] PostgreSQL connection setup is documented/configured for production-like deployment.
- [ ] Schema migration or schema initialization steps are documented clearly for the demo backend environment.
- [ ] Required backend production environment variables are documented, including secure secret handling and no reliance on the development secret in production.
- [ ] Object storage configuration is documented for public facility images and private reservation files, including expected access behavior.
- [ ] Worker/scheduler process configuration is documented or implemented for production deadline/completion jobs.
- [ ] A production-like smoke check can verify backend reachability and core health, including at least backend and database status.
- [ ] Existing behavior tests remain green, and any code changes are introduced using the repo's TDD workflow.

**Out of scope:**
- Frontend implementation.
- Vercel frontend deployment configuration until the frontend exists.
- Payment gateway integration.
- Full observability dashboards or advanced monitoring.
- Changing MVP reservation/domain behavior unless required for deployment readiness.
