## What to build

Add a current-user authentication endpoint so the frontend can discover the authenticated User Account through a clean API contract instead of probing role shell endpoints. The endpoint should return current user identity and role only; frontend routing remains responsible for mapping roles to landing pages.

## Acceptance criteria

- [ ] `GET /auth/me` is available for authenticated requests.
- [ ] A valid active User Account token returns the current user's `id`, `email`, `full_name`, `role`, and `is_active`.
- [ ] Missing, invalid, or expired tokens receive the same authentication behavior and Bahasa Indonesia error style as existing protected auth routes.
- [ ] Inactive User accounts cannot use the endpoint successfully.
- [ ] The endpoint does not return frontend landing routes or UI-specific navigation metadata.
- [ ] The endpoint uses the existing account/session/access patterns rather than decoding role information in frontend-facing token claims.
- [ ] Behavior tests cover valid token, missing token, invalid token, and inactive-account behavior through the FastAPI HTTP interface.
- [ ] API documentation or README endpoint listing is updated to include `GET /auth/me`.

## Agent Brief

**Category:** enhancement
**Summary:** Add a current-user authentication endpoint that returns the authenticated User Account identity through a clean API contract.

**Current behavior:**
Authenticated frontend routing currently has to infer the current User Account by probing role-specific shell endpoints. The backend already has session resolution behavior for bearer tokens and protected routes, but there is no role-neutral `GET /auth/me` endpoint for discovering the active authenticated user.

**Desired behavior:**
`GET /auth/me` should be available to authenticated requests and should return only the current User Account identity: `id`, `email`, `full_name`, `role`, and `is_active`. The endpoint should use the existing account/session/access patterns for resolving the active user from the bearer token. Missing, invalid, expired, or inactive-account tokens should receive the same authentication behavior and Bahasa Indonesia error style as existing protected auth routes. The response must not include frontend landing routes, route-selection hints, or UI navigation metadata.

**Key interfaces:**
- `UserAccount` or equivalent public account DTO — should provide the authenticated user fields returned by the endpoint without exposing password hashes or token internals.
- Existing current-user/session dependency — should remain the source of truth for token validation, expired or invalid token handling, and inactive-account blocking.
- Account/auth route registration — should expose `GET /auth/me` as a protected auth endpoint.
- API endpoint documentation or endpoint listing — should include `GET /auth/me`.

**Acceptance criteria:**
- [ ] `GET /auth/me` is available for authenticated requests.
- [ ] A valid active User Account token returns `id`, `email`, `full_name`, `role`, and `is_active`.
- [ ] Missing bearer credentials receive the same HTTP status and Bahasa Indonesia error style as existing protected auth routes.
- [ ] Invalid or expired bearer tokens receive the same HTTP status and Bahasa Indonesia error style as existing protected auth routes.
- [ ] Inactive User Account tokens cannot use the endpoint successfully.
- [ ] The endpoint response does not include frontend landing routes, route-selection hints, or UI navigation metadata.
- [ ] Behavior tests cover valid token, missing token, invalid token, and inactive-account behavior through the FastAPI HTTP interface.
- [ ] API documentation or the endpoint listing is updated to include `GET /auth/me`.

**Out of scope:**
- Adding frontend route mapping or navigation behavior.
- Adding role or landing-page claims to access tokens.
- Changing login, refresh, registration, or role shell endpoint behavior beyond any shared auth dependency reuse needed for this endpoint.
