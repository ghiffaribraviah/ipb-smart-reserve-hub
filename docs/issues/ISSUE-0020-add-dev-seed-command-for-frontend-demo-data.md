## What to build

Create a dev-only, idempotent seed command that gives frontend developers predictable local data for the Student Frontend Reservation Tracer. The command should seed local demo users, Facilities, Facility Categories, Facility Images, Facility Open Hours, Organization Units, Staff assignment, and public-calendar blocking Reservations without depending on test utilities or exposing an HTTP endpoint.

## Acceptance criteria

- [ ] A documented dev-only CLI seed command can be run locally, for example with `uv run python -m app.dev.seed` or an equivalent command.
- [ ] The seed command refuses to run when the app is configured for production.
- [ ] The command is idempotent: running it repeatedly does not duplicate users, Facilities, Organization Units, assignments, Open Hours, images, or blocking Reservations.
- [ ] Fixed local-only demo credentials are created and documented for one Student User account, one Staff User account, and one Super Admin User account.
- [ ] The demo Student account starts with a clean Reservation list.
- [ ] A separate synthetic Student User account owns seeded blocking Reservations used for public calendar display.
- [ ] Seeded data includes at least two or three active Facilities with categories, public cover/detail images, open-hours summaries, Facility Open Hours, active Organization Units, and one Staff assignment.
- [ ] Seeded data includes at least one approved and one pending/held future blocking Reservation for public calendar visibility.
- [ ] The seed code lives outside `tests/` and uses normal application database/session setup rather than `tests.DataBuilder`.
- [ ] Behavior tests verify the public seed command/module behavior, production refusal, and idempotent database-observable outcomes.
- [ ] README or local development docs include the seed command and demo credentials.

## Agent Brief

**Category:** enhancement
**Summary:** Add a dev-only, idempotent seed command that creates predictable local/demo data for the Student Frontend Reservation Tracer.

**Current behavior:**
The backend has domain models and workflows for users, Facilities, Facility Categories, Facility Images, Facility Open Hours, Organization Units, staff Facility assignments, and Reservations, but there is no public dev seed command that frontend developers can run to populate local data. Local demo data must currently be assembled manually or through test-only helpers, which is unsuitable for frontend/manual QA workflows.

**Desired behavior:**
A documented development-only command should seed predictable local data through the normal application database/session setup. The command must refuse to run when the app is configured for production. It must be safe to run repeatedly without duplicating seeded users, Facilities, Organization Units, staff assignments, Facility Open Hours, Facility Images, or blocking Reservations.

The seeded data should include fixed local-only credentials for one Student User account, one Staff User account, and one Super Admin User account. The demo Student account should start with no Reservations. Public-calendar blocking Reservations should belong to a separate synthetic Student User account, so they are visible to Facility calendar readers without polluting the demo Student workflow.

Seeded Facility data should be realistic enough for frontend reservation tracing: at least two or three active Facilities, active categories, public cover/detail images, open-hours summaries, Facility Open Hours, active Organization Units, one Staff assignment, and at least one approved and one pending/held future blocking Reservation.

**Key interfaces:**
- Public seed command/module interface, for example `uv run python -m app.dev.seed` or an equivalent documented command.
- Application Settings / environment handling — the seed command must detect production configuration and fail closed when `IPB_ENVIRONMENT` is production.
- Normal database/session setup — seed code should use the application database/session infrastructure, not test-only fixture helpers.
- `User` / User account workflow contract — seeded users must have valid password hashes, roles, active status, and fixed local-only credentials.
- Facility catalog data contracts — seeded Facilities, categories, images, open-hours summaries, and open hours should be observable through existing public Facility browsing/calendar behavior.
- Reservation persistence / public calendar contract — seeded blocking Reservations must use statuses that are treated as public-calendar blocking entries.
- Local development documentation — README or equivalent docs must list the command and demo credentials.
