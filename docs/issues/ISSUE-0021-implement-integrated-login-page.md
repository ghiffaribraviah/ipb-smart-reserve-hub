## Original Issue Body

## What to build

Build the `/login` experience as a fully integrated authentication slice. Users should enter campus credentials, submit to the backend login endpoint, have the access token stored through the frontend auth session abstraction, load their current identity, and land in the correct role area.

The page should follow the supplied login reference direction while staying implementation-ready: real HTML/CSS form controls, a styled text `IPB SRH` wordmark, a desktop split layout with a temporary right-side visual placeholder, and a mobile form-first layout. The forgot-password affordance should stay hidden until a recovery workflow exists.

## Acceptance criteria

- [ ] `/login` renders the IPB Smart Reserve Hub login form with email and password fields, submit button, footer links, and responsive desktop/mobile layout.
- [ ] Submitting valid credentials calls `POST /auth/login`, stores the returned access token via `authSession`, calls `GET /auth/me`, and redirects by role: `student` to `/student`, `staff` to `/staff`, and `super_admin` to `/admin`.
- [ ] Existing stored tokens are verified with `GET /auth/me`; valid tokens redirect by role, while invalid tokens are cleared and the login form is shown.
- [ ] Client-side validation is minimal: email is required and email-shaped, password is required, and backend remains source of truth for credential or inactive-account errors.
- [ ] Backend `detail` messages are shown directly in a compact form error state, and editing either field clears the displayed error.
- [ ] While login is submitting, the form prevents repeat submission, keeps field values visible, and changes the submit button label to `Signing in...`.
- [ ] Minimal `/student`, `/staff`, and `/admin` landing placeholders exist so successful role redirects resolve cleanly without implementing full dashboards.
- [ ] Routed frontend behavior tests cover successful student login, backend credential errors, valid stored-token redirect, invalid stored-token clearing, and required-field validation.
- [ ] The login page is visually checked at desktop and mobile viewport sizes with no text overlap, clipped controls, unreadable contrast, or blank critical UI.
