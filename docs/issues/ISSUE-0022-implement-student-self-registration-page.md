## Original Issue Body

## What to build

Implement unauthenticated student self-registration at `/register` so a student can create an account with institutional identity details, then sign in separately through the existing login flow.

The slice should use the auth split-screen visual language, submit the backend registration contract to `POST /auth/register`, show validation and backend errors, and redirect to `/login` with a success message after account creation.

## Acceptance criteria

- [ ] `/register` renders a student registration form using the auth split-screen layout on desktop and a single-column form on mobile.
- [ ] The form collects full name, NIM, phone number, email address, password, and password confirmation.
- [ ] The client validates required fields, email shape, minimum 8-character password length, and matching password confirmation before submitting.
- [ ] The submitted request calls `POST /auth/register` with `email`, `password`, `full_name`, `nim`, and `phone`, without sending `confirmPassword`.
- [ ] Backend registration errors are shown in-place without storing an auth token or redirecting.
- [ ] Successful registration redirects to `/login` with a success message and does not auto-login the student.
- [ ] The registration behavior is covered by frontend route-level tests and verified visually on desktop and mobile.

- `/register` public route — renders the student registration experience.
- `POST /auth/register` — accepts `email`, `password`, `full_name`, `nim`, and `phone`.
- `/login` public route — displays a success message after registration redirect.
- Auth session storage — remains empty after successful registration until the student signs in.
