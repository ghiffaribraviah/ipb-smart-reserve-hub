Add the public Facility Category contract needed by student facility discovery. Students and frontend integrations should be able to load active facility categories with stable slugs, icon hints, and active facility counts for home shortcuts and catalog filters.

## Agent Brief

**Category:** enhancement
**Summary:** Add a public Facility Category contract with stable slugs, icon hints, and active facility counts for student discovery.

**Current behavior:**
Facility Categories exist as managed records for grouping Facilities, but public facility discovery does not expose a dedicated category collection. Public facility responses currently identify the category by display name, and frontend category shortcuts/catalog filters do not have a backend-owned stable slug contract. Facility Category persistence does not yet include the public slug and icon hint fields required by the planned student frontend.

**Desired behavior:**
Public clients should be able to call `GET /facility-categories` without authentication and receive the active Facility Categories used for student browsing. Each response item should include the category `id`, display `name`, stored `slug`, optional `icon_hint`, and `facility_count`. The count must include only active Facilities visible in the Facility Catalog. Active categories with no active Facilities should still be returned with `facility_count: 0`, while inactive categories should be hidden from the public endpoint.

Facility Category slugs should be backend-owned, unique, non-null, and stored rather than derived ad hoc in each response. Existing seed and test categories should receive deterministic slugs and sensible icon hints, or null/default hints where no specific hint is known.

**Key interfaces:**
- `GET /facility-categories` public HTTP contract — returns active category metadata for student home shortcuts and catalog filters.
- Facility Category persistence model — stores a unique non-null public `slug` and optional `icon_hint` alongside the display name and active flag.
- Facility Catalog/category read boundary — computes public category projections and active Facility counts without exposing inactive Facilities.
- Seed/test data setup — creates deterministic category slugs and icon hints so tests and local development data are stable.
- Project documentation — records the public category contract and removes the stale backend gap for category slugs where appropriate.

**Acceptance criteria:**
- [ ] `GET /facility-categories` returns HTTP 200 for unauthenticated clients.
- [ ] The response contains active Facility Categories only.
- [ ] Each category item includes `id`, `name`, stored `slug`, optional `icon_hint`, and `facility_count`.
- [ ] `facility_count` counts only active Facilities visible in the Facility Catalog.
- [ ] Active categories with zero active Facilities are returned with `facility_count: 0`.
- [ ] Inactive categories are not returned by the public endpoint.
- [ ] Facility Category persistence includes a unique, non-null backend-owned slug.
- [ ] Facility Category persistence supports an optional icon hint.
- [ ] Existing seed and test categories receive deterministic slugs and sensible icon hints or null/default hints.
- [ ] Public HTTP/API behavior is covered by tests, with any useful service-level tests added around the category projection/counting rules.
- [ ] `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` are updated for the category contract.

**Out of scope:**
- Implementing facility collection pagination, search, sorting, featured ranking, or category filtering beyond what is required to count active public Facilities.
- Adding or changing frontend screens.
- Building Facility Category management UI.
- Introducing a migration framework such as Alembic.
- Adding manual featured facility curation.
- Changing Reservation, Payment, Approval Letter, or User Account contracts.
