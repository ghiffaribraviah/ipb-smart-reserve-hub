## What to build

Convert the public Facility Catalog collection into the frontend integration contract: a paginated facility response envelope with keyword search, category slug filtering, minimum capacity filtering, explicit sorting, and page metadata.

## Agent Brief

**Category:** enhancement
**Summary:** Convert the public Facility Catalog collection into a paginated, filterable, sortable frontend integration contract.

**Current behavior:**
`GET /facilities` is public and returns active Facility Catalog items as an unpaginated list. Existing catalog items expose the public comparison fields the student frontend needs, including facility identity, name, location, capacity, category display name, cover image URL, review summary fields, price summary, and open-hours summary. The collection does not currently expose pagination metadata or documented query parameters for keyword search, category slug filtering, minimum capacity filtering, or explicit sorting.

**Desired behavior:**
Public clients should be able to call `GET /facilities` without authentication and receive a paginated response envelope. The envelope should include `items`, `page`, `page_size`, `total_items`, and `total_pages`. Existing Facility Catalog item fields should remain available inside `items`.

The endpoint should support optional query parameters for keyword search, category slug filtering, minimum capacity filtering, sorting, page number, and page size. Defaults should keep the first page easy to consume: page `1`, page size `12`, and maximum page size `50`. Unknown sort values should fail validation instead of silently falling back.

**Key interfaces:**
- `GET /facilities` public HTTP contract — changes from a bare array response to a paginated envelope with filtering, sorting, and page metadata.
- Facility Catalog collection response type — preserves existing item fields inside `items` while adding pagination fields around them.
- Facility Catalog query input — accepts `q`, `category`, `min_capacity`, `sort`, `page`, and `page_size` with validation and documented defaults.
- Facility Catalog read boundary — loads only active public Facilities and applies the public filtering/sorting/pagination rules without exposing private staff, reservation, payment, or student data.
- Facility Category slug contract — category filtering uses the stored backend-owned Facility Category slug, not display-name matching.
- Project documentation — records the updated public facility collection contract and removes or resolves the stale backend gap for catalog filtering/pagination.

**Acceptance criteria:**
- [ ] `GET /facilities` returns HTTP 200 for unauthenticated clients.
- [ ] `GET /facilities` returns an envelope with `items`, `page`, `page_size`, `total_items`, and `total_pages`.
- [ ] Existing Facility Catalog item fields remain available for each entry inside `items`.
- [ ] The default page is `1` when `page` is omitted.
- [ ] The default page size is `12` when `page_size` is omitted.
- [ ] The maximum page size is `50`.
- [ ] `q` filters active Facilities by case-insensitive contains matching across facility name and location.
- [ ] `category` filters active Facilities by stored Facility Category slug.
- [ ] `min_capacity` filters active Facilities whose capacity is greater than or equal to the requested value.
- [ ] `sort=name_asc` sorts by facility name ascending.
- [ ] `sort=capacity_desc` sorts by capacity descending.
- [ ] `sort=rating_desc` sorts by public rating descending.
- [ ] `sort=price_asc` sorts by facility price ascending.
- [ ] `sort=price_desc` sorts by facility price descending.
- [ ] Unknown `sort` values produce a validation error instead of silently falling back.
- [ ] Public HTTP/API behavior is covered by tests.
- [ ] Facility Catalog service/read-boundary behavior is covered by focused tests where useful.
- [ ] `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` document the changed facility collection contract.

**Out of scope:**
- Changing Facility detail, public calendar, availability, or reservation-time-selection contracts.
- Adding or changing frontend screens.
- Adding Facility Category management UI.
- Changing the public Facility Category collection beyond using stored slugs for filtering.
- Implementing featured/explorable facility ranking for the student home page.
- Exposing private staff, reservation, payment, or student data in Facility Catalog responses.
