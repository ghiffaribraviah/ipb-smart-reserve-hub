## Agent Brief

**Category:** enhancement
**Summary:** Add a featured Facility Catalog query for the student home page using the existing public paginated facility collection envelope.

**Current behavior:**
`GET /facilities` is public and returns active Facility Catalog items in a paginated envelope with `items`, `page`, `page_size`, `total_items`, and `total_pages`. The collection already supports `q`, `category`, `min_capacity`, `sort`, `page`, and `page_size`. Catalog items expose the public comparison fields needed by the frontend, including facility identity, name, location, capacity, category, cover image URL, public rating average, review count, price summary, and open-hours summary.

The student home page still has a backend gap for a compact featured/explorable facility query. `docs/frontend/backend-gaps.md` currently lists this as blocking for integration.

**Desired behavior:**
Public clients should be able to call `GET /facilities?featured=true` and receive the same standard paginated facility envelope. The response should contain active Facilities only and should remain unauthenticated.

When `featured=true` is present, the query should rank Facilities by useful public signals without introducing manual curation for MVP: Facilities with an active cover image first, then higher visible review count, then higher visible rating average, then facility name ascending. The query should accept `limit` as an alias for the featured page size so the student home page can request a compact list such as `GET /facilities?featured=true&limit=8`.

**Key interfaces:**
- `GET /facilities` public HTTP contract — add optional `featured=true` mode while preserving the standard paginated response envelope.
- Facility Catalog query input — accepts `featured` and `limit`; `limit` acts as the featured page-size alias.
- Facility Catalog service/read boundary — continues to load active public Facilities only and derives featured ranking from public image/review signals.
- Facility Catalog item response — preserve existing fields inside `items`.
- Project documentation — update `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` to document the featured query behavior and resolve the student-home backend gap.

**Acceptance criteria:**
- [ ] `GET /facilities?featured=true` returns HTTP 200 for unauthenticated clients.
- [ ] `GET /facilities?featured=true` returns the standard paginated envelope with `items`, `page`, `page_size`, `total_items`, and `total_pages`.
- [ ] Existing Facility Catalog item fields remain available for each entry inside `items`.
- [ ] `limit` is accepted as an alias for the featured page size.
- [ ] Featured ranking prefers Facilities with an active cover image.
- [ ] Within the cover-image grouping, featured ranking prefers higher visible review count.
- [ ] Within matching cover/review-count grouping, featured ranking prefers higher visible rating average.
- [ ] Remaining ties are sorted by facility name ascending for deterministic output.
- [ ] Featured query returns active Facilities only.
- [ ] Featured query remains public and unauthenticated.
- [ ] No manual featured flag, display priority field, or admin curation UI is introduced.
- [ ] Public HTTP/API tests cover featured ranking, `limit` behavior, active-only filtering, public access, and envelope shape.
- [ ] Facility Catalog service tests cover featured ranking where useful.
- [ ] `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` document the featured query behavior.

**Out of scope:**
- Adding a manual featured flag, display priority field, or admin curation UI.
- Adding a separate `/facilities/featured` route unless implementation constraints make it necessary; the requested contract is `GET /facilities?featured=true`.
- Changing Facility detail, public calendar, availability, or reservation-time-selection contracts.
- Changing frontend screens in this issue.
- Exposing private staff, reservation, payment, or student data.

**Implementation notes:**
The current route is in `backend/app/api/routes/facility_routes.py`, and the current catalog behavior/ranking helpers are in `backend/app/services/facilities.py`. Existing tests in `backend/tests/test_facility_browsing.py` cover the current paginated collection and are the natural place for the new public HTTP behavior tests.
