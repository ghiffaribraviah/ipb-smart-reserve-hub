## Agent Brief

**Category:** enhancement
**Summary:** Add structured extra requirements to student Facility Reservation submission and student reservation responses.

**Current behavior:**
Student reservation submission accepts the core reservation details only: activity, description, participant count, organization unit, contact phone, and time range. Student reservation list and detail responses return the main reservation fields and lifecycle deadlines, but they do not expose structured operational requirements such as AV support, logistics coordination, extra cleaning, security personnel, or notes.

**Desired behavior:**
Students can optionally submit an `extra_requirements` object when creating a Facility Reservation. The object should contain boolean flags for `av_support`, `logistics_coordination`, `extra_cleaning`, and `security_personnel`, plus optional free-text `notes`. The flags should be persisted as explicit Reservation fields, with notes stored separately, and student reservation creation/list/detail responses should expose the same information as a nested `extra_requirements` object.

Omitting `extra_requirements` must remain backward-tolerant: existing clients and tests that do not send the object should still create reservations successfully, with all flags defaulting to `false` and notes represented consistently as null or an empty string according to the implementation decision. Existing availability, conflict, organization unit, facility access, lifecycle, document, payment, and cancellation rules should continue to apply unchanged.

**Key interfaces:**
- `ReservationSubmission` or equivalent reservation creation input type — accept optional structured extra requirements.
- Reservation persistence model/schema — add explicit fields for the four boolean requirement flags and a separate notes field.
- Student reservation projection/response type — include nested `extra_requirements` in creation, list, and detail responses.
- Reservation submission HTTP request schema — accept the optional nested object while preserving existing request compatibility.
- Test data builders/seed helpers — support the new fields with deterministic defaults so existing tests remain clear.
- Project documentation — document the reservation extra requirements contract in README, domain context, and frontend backend-gap notes.

**Acceptance criteria:**
- [ ] `POST /facilities/{facility_id}/reservations` accepts `extra_requirements` with `av_support`, `logistics_coordination`, `extra_cleaning`, `security_personnel`, and optional `notes`.
- [ ] Submitted extra requirement flags and notes are persisted on the Reservation record as explicit fields.
- [ ] Requests that omit `extra_requirements` still succeed and default all flags to `false`, with notes consistently null or empty.
- [ ] Reservation creation response exposes nested `extra_requirements` with the persisted values.
- [ ] `GET /student/reservations` exposes nested `extra_requirements` for each reservation.
- [ ] `GET /student/reservations/{reservation_id}` exposes nested `extra_requirements` for the reservation detail.
- [ ] Existing reservation availability, conflict, organization unit, access, lifecycle, document, payment, and cancellation behavior remains unchanged.
- [ ] Behavior is covered through student reservation submission HTTP/API tests and service-level tests where useful, following the repo TDD workflow.
- [ ] Seed/test data builders are updated so existing reservation tests remain deterministic and readable.
- [ ] README, CONTEXT, and frontend backend-gap documentation describe the extra requirements behavior.

**Out of scope:**
- Frontend implementation or UI changes.
- Staff/admin filtering, reporting, or operational queues for extra requirements.
- New ReservationStatus values or lifecycle transitions.
- Changing existing document, payment, cancellation, availability, conflict, or access rules.
- Introducing a migration framework or broader database deployment architecture.
