## What to build

Enrich student reservation list/detail responses with explicit document, payment, and rejection projections so the frontend can route accurately between upload-needed, waiting-review, approved, declined, accepted, completed, and detail states without adding new ReservationStatus values.

## Acceptance criteria

- [ ] Existing ReservationStatus values remain unchanged; no UI substate statuses are added.
- [ ] Student reservation responses include a document projection with generated approval letter metadata when present, signed approval letter metadata when present, document review status, and document rejection reason when relevant.
- [ ] Student reservation responses include a payment projection with `required`, receipt metadata when present, payment review status, and payment rejection reason when relevant.
- [ ] Student reservation responses include a rejection projection with `source` and `reason` for terminal rejected reservations.
- [ ] A nullable Reservation rejection source enum is persisted with values `document` and `payment`.
- [ ] Document review rejection records `rejection_source: document` and payment review rejection records `rejection_source: payment`.
- [ ] Rejected rows without a persisted rejection source are exposed as `unknown`.
- [ ] Cancellation rejection remains separate and does not set terminal rejection source.
- [ ] Missing document/payment metadata is represented as absent or null according to the response schema, without fake filenames or dates.
- [ ] Existing staff document/payment review workflows and responses keep working.
- [ ] Behavior is covered by student reservation list/detail tests, approval letter workflow tests, payment workflow tests, and legacy unknown-source tests.
- [ ] `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` document workflow projections and rejection source behavior.

## Agent Brief

**Category:** enhancement
**Summary:** Enrich student reservation list/detail responses with workflow projections for document, payment, and terminal rejection state without adding new reservation statuses.

**Current behavior:**
Student reservation list/detail responses expose the coarse reservation status and deadline fields, but they do not expose enough document, payment, or rejection context for the frontend to route accurately through the planned student workflow pages.

Approval letter, signed approval letter, and payment receipt records already exist. Document and payment review workflows can both transition a reservation to `rejected` and store a rejection reason, but the student-facing reservation response cannot tell whether the terminal rejection came from document review, payment review, or legacy data with no source recorded.

Cancellation rejection is already represented separately from terminal reservation rejection and should remain separate.

**Desired behavior:**
Student reservation list and detail responses should include explicit nested projections for document workflow state, payment workflow state, and terminal rejection state.

The projections should let the frontend distinguish:
- approval letter generated vs absent
- signed approval letter uploaded vs absent
- document upload needed vs document waiting for review vs document rejected
- payment required vs not required
- payment receipt uploaded vs absent
- payment upload needed vs payment waiting for review vs payment rejected
- approved, accepted/detail, completed, and terminal rejection detail states

Existing `ReservationStatus` values must remain unchanged. UI-only substates should be derived from the new projections, not added to the status enum.

A nullable persisted reservation rejection source should record whether terminal rejection came from document review or payment review. Existing rejected rows with no persisted source should be exposed to students as `unknown`. Cancellation rejection must not set this terminal rejection source.

Missing document/payment metadata should be represented as absent or null according to the response schema. Do not synthesize fake filenames, upload times, generated times, receipt metadata, or rejection reasons.

**Key interfaces:**
- `ReservationStatus` — keep the existing values unchanged.
- Student reservation list/detail response shape — add document, payment, and rejection projections to the public student API contract.
- Reservation persistence model — add a nullable rejection source enum with persisted values for document and payment rejection.
- Document review workflow — when staff reject a signed approval letter, persist terminal rejection source as document and expose the document rejection reason through the projections.
- Payment review workflow — when staff reject a payment receipt, persist terminal rejection source as payment and expose the payment rejection reason through the projections.
- Cancellation workflow — keep cancellation rejection reason separate and do not write terminal rejection source.
- Existing staff document/payment review responses — preserve existing behavior while adding the new persisted/source data.
- Project documentation — update README, CONTEXT, and frontend backend gap docs to describe workflow projections and rejection source behavior.

**Acceptance criteria:**
- [ ] Existing `ReservationStatus` values remain unchanged; no UI substate statuses are added.
- [ ] Student reservation list responses include document projection data with generated approval letter metadata when present, signed approval letter metadata when present, document review status, and document rejection reason when relevant.
- [ ] Student reservation detail responses include the same document projection behavior as list responses.
- [ ] Student reservation list responses include payment projection data with `required`, receipt metadata when present, payment review status, and payment rejection reason when relevant.
- [ ] Student reservation detail responses include the same payment projection behavior as list responses.
- [ ] Student reservation list/detail responses include a rejection projection with `source` and `reason` for terminal rejected reservations.
- [ ] A nullable reservation rejection source enum is persisted with values `document` and `payment`.
- [ ] Document review rejection records `rejection_source: document`.
- [ ] Payment review rejection records `rejection_source: payment`.
- [ ] Rejected rows without a persisted rejection source are exposed as `unknown`.
- [ ] Cancellation rejection remains separate and does not set terminal rejection source.
- [ ] Missing document/payment metadata is represented as absent or null according to the response schema, without fake filenames or dates.
- [ ] Existing staff document review workflows and responses keep working.
- [ ] Existing staff payment review workflows and responses keep working.
- [ ] Behavior is covered by student reservation list/detail tests, approval letter workflow tests, payment workflow tests, and legacy unknown-source tests.
- [ ] README, CONTEXT, and frontend backend gap docs document workflow projections and rejection source behavior.

**Out of scope:**
- Adding new reservation status values.
- Implementing frontend routing or UI pages.
- Adding student-facing document or payment receipt download support beyond metadata projection.
- Changing staff review authorization rules.
- Changing cancellation semantics or merging cancellation rejection with terminal reservation rejection.
