## Agent Brief

**Category:** enhancement
**Summary:** Add student-owned private download endpoints for uploaded signed approval letters and payment receipts.

**Current behavior:**
Students can upload signed approval letters and payment receipts, and staff assigned to the reservation facility can download those uploaded private files for review. Students can also download the generated approval letter that belongs to their reservation.

Student reservation list/detail responses already expose workflow projections for generated approval letter metadata, signed approval letter metadata, payment receipt metadata, document review status, payment review status, and terminal rejection source/reason. These projections let the frontend know whether uploaded file metadata exists, but there is no student-owned API path for reopening the uploaded signed approval letter or uploaded payment receipt itself.

Uploaded signed approval letters and payment receipts are private reservation files. They must only be served after backend authorization verifies the authenticated student owns the reservation.

**Desired behavior:**
Authenticated students should be able to download their own uploaded signed approval letter and uploaded payment receipt through student-owned endpoints.

Each download should return the stored file bytes, the stored content type, and a `Content-Disposition` attachment filename matching the uploaded filename. If the reservation does not belong to the student, the API should use the existing student reservation not-found/ownership behavior. If the uploaded file is missing, the API should use the existing project-consistent unavailable or not-found behavior for that file type.

Existing staff-owned private file download endpoints must keep their current behavior unchanged.

Student reservation document/payment projections should expose enough metadata or action availability for the frontend to show download actions only when uploaded files exist. Missing uploaded files should continue to be represented as null metadata, not fake filenames or dates.

**Key interfaces:**
- Student reservation private-file downloads — add student-owned download behavior for uploaded signed approval letters and payment receipts.
- Student reservation ownership checks — downloads must authorize through student ownership of the reservation, not through staff review access.
- Private storage access — downloads should read the stored object for the uploaded signed approval letter or receipt and return the persisted bytes and content type.
- Student reservation document projection — signed approval letter metadata/action availability should indicate when the uploaded signed letter can be downloaded.
- Student reservation payment projection — receipt metadata/action availability should indicate when the uploaded payment receipt can be downloaded.
- Staff private-file downloads — existing staff signed approval letter and payment receipt download behavior should remain unchanged.
- Project documentation — update README, CONTEXT, and frontend backend gap docs to describe the new student-owned download endpoints and frontend-facing availability.

**Acceptance criteria:**
- [ ] Students can download their uploaded signed approval letter through a student-owned endpoint.
- [ ] Students can download their uploaded payment receipt through a student-owned endpoint.
- [ ] Each student download returns the stored file bytes.
- [ ] Each student download returns the stored content type.
- [ ] Each student download returns a `Content-Disposition` attachment filename based on the uploaded filename.
- [ ] Another student cannot access a signed approval letter uploaded for someone else's reservation.
- [ ] Another student cannot access a payment receipt uploaded for someone else's reservation.
- [ ] Missing uploaded signed approval letters return the existing project-consistent missing/unavailable error behavior.
- [ ] Missing uploaded payment receipts return the existing project-consistent missing/unavailable error behavior.
- [ ] Existing staff signed approval letter download behavior remains unchanged.
- [ ] Existing staff payment receipt download behavior remains unchanged.
- [ ] Student reservation document/payment projections expose enough metadata or action availability for the frontend to show download actions only when uploaded files exist.
- [ ] Behavior is covered by HTTP/API tests for owner access, other-student denial, missing uploaded files, content type, content disposition, and file bytes for both uploaded file types.
- [ ] README, CONTEXT, and frontend backend gap docs document the new student-owned download endpoints.

**Out of scope:**
- Changing upload validation rules, allowed file types, or file size limits.
- Changing staff review authorization rules.
- Changing generated approval letter download behavior except where shared response helpers are reused.
- Implementing frontend UI for the new download actions.
- Replacing the current private storage implementation with production object storage or signed URLs.
