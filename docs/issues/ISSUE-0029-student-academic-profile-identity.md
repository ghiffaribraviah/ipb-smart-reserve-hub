## What to build

Extend current-user identity for students so `/auth/me` returns NIM, phone, and a best-effort academic profile derived from NIM. The backend remains the source of truth for derivation, and unknown NIM formats should not block account access.

## Acceptance criteria

- [ ] Student current-user responses include `nim` and `phone` from the User account.
- [ ] Student current-user responses include `academic_profile` with `program_studi`, `faculty`, `entry_year`, and `degree`.
- [ ] A deep Academic Profile Deriver module maps known NIM prefixes/year information into academic profile fields.
- [ ] Unknown or unparsable NIM values return null academic profile fields instead of rejecting registration, login, refresh, or `/auth/me`.
- [ ] Registration remains lightly validated and does not require successful academic derivation.
- [ ] Staff and Super Admin identity responses remain sensible and do not expose student-only fields incorrectly.
- [ ] Behavior is covered by current-user HTTP/API tests and isolated Academic Profile Deriver service tests.
- [ ] `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` document the student profile identity contract and best-effort derivation behavior.

## Agent Brief

**Category:** enhancement
**Summary:** Extend current-user identity responses with student NIM, phone, and best-effort academic profile derivation.

**Current behavior:**
The current-user endpoint returns only the shared account identity fields: account ID, email, full name, role, and active state. Student registration captures NIM and phone on the User account, but those student profile fields are not exposed by current-user identity responses. The frontend student profile page therefore cannot use the backend as the source of truth for NIM, phone, program study, faculty, entry year, or degree.

**Desired behavior:**
When the authenticated current user is a student, the current-user response should include the stored `nim` and `phone` from the User account plus an `academic_profile` object. The academic profile should contain `program_studi`, `faculty`, `entry_year`, and `degree`, derived by backend-owned best-effort NIM parsing.

Known NIM formats should map to meaningful academic profile fields. Unknown, missing, or unparsable NIM values should not reject registration, login, token refresh, or current-user lookup. Instead, the response should remain successful and academic profile fields that cannot be derived should be `null`.

Staff and Super Admin current-user responses should remain sensible and should not expose student-only account data as if it applied to those roles. If the response shape includes student-only fields for all roles, non-student values should be explicitly null rather than misleading.

**Key interfaces:**
- `GET /auth/me` response contract — should expose student identity fields needed by the profile frontend while preserving the existing account identity fields.
- Public current-user account representation — should carry stored student NIM and phone when the role is student.
- Academic Profile Deriver service/module — should own NIM parsing and mapping for `program_studi`, `faculty`, `entry_year`, and `degree` so HTTP routes do not derive academic fields inline.
- Student registration and authentication flows — should remain lightly validated and should not depend on successful academic profile derivation.
- Project documentation — should describe the student profile identity contract and best-effort derivation behavior.

**Acceptance criteria:**
- [ ] Student `/auth/me` responses include `nim` and `phone` from the User account.
- [ ] Student `/auth/me` responses include `academic_profile.program_studi`, `academic_profile.faculty`, `academic_profile.entry_year`, and `academic_profile.degree`.
- [ ] Known NIM prefixes and year information are mapped by an isolated Academic Profile Deriver module/service, with isolated service tests covering known mappings.
- [ ] Unknown, missing, or unparsable NIM values return successful auth/current-user responses with null academic profile fields instead of blocking registration, login, refresh, or `/auth/me`.
- [ ] Registration remains lightly validated and does not require successful academic derivation.
- [ ] Staff and Super Admin `/auth/me` responses do not misleadingly expose student-only NIM, phone, or academic profile values.
- [ ] Behavior is covered by current-user HTTP/API tests and isolated Academic Profile Deriver service tests.
- [ ] `README.md`, `CONTEXT.md`, and `docs/frontend/backend-gaps.md` document the student profile identity contract and best-effort derivation behavior.

**Out of scope:**
- Student profile editing.
- Strict admissions-grade NIM validation or complete IPB academic code coverage.
- Frontend implementation or frontend integration work.
- Token expiry metadata, auto-login after registration, or unrelated auth contract changes.
- Introducing a migration framework or changing persistence architecture beyond what this contract requires.
